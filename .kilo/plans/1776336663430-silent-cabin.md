# Plan: Implement Sequential Bot Interaction for Deep Links with ?start= Parameter

## Goal
Implement feature where bot:
1. Scans messages one-by-one in a specified ID range (e.g., 1000-1200).
2. For each message, checks for deep links containing `?start=XXXX`.
3. If found, immediately sends `/start XXXX` to the corresponding bot.
4. Continues to next message ID sequentially.
5. Logs all interactions and handles errors.

This enables real-time, message-by-message bot interaction via deep links.

## Current State Analysis
- **Scraping**: Currently batch-scrapes messages in chunks, extracts links afterward.
- **Interaction**: No bot interaction capability.
- **Range**: Supports start/end ID, but processes in batches.
- **Constraints**: Telegram rate limits, need sequential processing with delays.

## Proposed Changes
### 1. Modify Scraping to Sequential Processing
- Change `ScrapingService.scrapeChannel()` to process messages one-by-one in range.
- For each message, immediately check and interact with bots if links found.

### 2. Integrate Bot Interaction
- Add `BotInteractionService` for sending `/start` commands.
- Call interaction immediately after link extraction per message.

### 3. Update Workflow
- User specifies range (startId, endId).
- Bot iterates: fetch message ID, check links, send /start, delay, next ID.

### 4. Error Handling and Logging
- Log per-message checks and interactions.
- Handle timeouts, invalid bots, rate limits.

## Implementation Steps
### Phase 1: Sequential Scraping
1. Update `ScrapingService.scrapeChannel(channel, startId, endId)`:
   - Loop from startId to endId.
   - For each ID, fetch single message.
   - Extract ?start= links, parse botUsername and startParam.
   - Immediately call `botInteractionService.interact(botUsername, startParam)`.
   - Delay between fetches/interactions.

### Phase 2: Bot Interaction Service
2. Create `BotInteractionService`:
   - Send `/start param` to bot, wait for response, log.

### Phase 3: UI Updates
3. Update UI for range input and real-time logging.

### Phase 4: Testing
4. Test with sample ranges, verify sequential processing.

## Risks and Mitigations
- **Rate Limits**: Longer delays (3-5s per interaction).
- **Performance**: Slower than batch, but real-time.
- **Errors**: Robust error handling per message.

## Success Criteria
- Process range sequentially, send /start per valid link.
- Log all actions, handle errors gracefully.
- No crashes, respect limits.
- Performance: <10s per message average.

## Open Questions
- How to handle multiple links per message?
- Should interactions be deduplicated per bot?
- UI feedback for long ranges?
