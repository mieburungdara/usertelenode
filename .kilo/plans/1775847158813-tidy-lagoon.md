# Plan: Create History Module for Scraping Data

## Objective
Create a `history.js` module that stores information about channels and the last scraped ID range to enable resuming scrapes from the last position and track scraping history.

## Current State Analysis
- `deepLinkScraper.js` performs deep link scraping from Telegram channels
- It scrapes messages in ID ranges (startId to endId)
- No persistent storage of scraping progress or history
- Report generation happens but data is not saved for future use
- Users have to manually track which ranges they've already scraped

## Proposed Solution
Create a dedicated history module that:
- Stores channel scraping history in JSON format
- Tracks last scraped ID for each channel
- Maintains scraping sessions with timestamps
- Provides functions to retrieve and update history

## Implementation Plan

### 1. Create History Module (`utils/scrapingHistory.js`)
**Functions to implement:**
- `loadHistory()` - Load history from `scraping_history.json`
- `saveHistory(data)` - Save history to file
- `getLastScrapedId(channel)` - Get last scraped message ID for a channel
- `getChannelHistory(channel)` - Get complete history for a channel
- `updateHistory(channel, startId, endId, linksFound, stopped)` - Update history after scraping

**Data Structure:**
```json
{
  "channels": {
    "channel_identifier": {
      "channelName": "string",
      "lastScrapedId": number,
      "lastScrapedAt": "ISO timestamp",
      "totalScraped": number,
      "scrapingSessions": [
        {
          "startId": number,
          "endId": number,
          "scrapedAt": "ISO timestamp",
          "linksFound": number,
          "stopped": boolean
        }
      ]
    }
  }
}
```

### 2. Integrate History into Deep Link Scraper
**Modifications to `handlers/deepLinkScraper.js`:**
- Add import: `const { updateHistory } = require('../utils/scrapingHistory');`
- Store `parsedChannelId` in `reportData` for access in update calls
- Call `updateHistory()` after `generateReport()` in both completion and early-stop scenarios

### 3. File Structure
- New file: `utils/scrapingHistory.js`
- Modified file: `handlers/deepLinkScraper.js`
- New data file: `scraping_history.json` (created automatically)

## Benefits
- **Resume capability**: Users can continue scraping from last position
- **Progress tracking**: See which channels have been scraped and when
- **History analysis**: Review past scraping sessions
- **Data persistence**: No loss of progress between runs

## Potential Enhancements (Future)
- Add option to resume from last scraped ID + 1
- Include scraping statistics (success rate, time taken)
- Support for multiple concurrent scraping sessions
- Cleanup old history entries

## Testing Plan
- Unit tests for history module functions
- Integration test with deepLinkScraper
- Verify history persistence across restarts
- Test with different channel identifiers (username, ID)

## Dependencies
- Node.js fs module (already available)
- JSON for data storage (built-in)

## Risk Assessment
- **Low risk**: Module is self-contained, doesn't affect existing functionality
- **Data integrity**: JSON parsing errors handled gracefully
- **File permissions**: Standard file operations, no special permissions needed
